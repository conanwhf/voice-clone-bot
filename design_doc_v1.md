# VoiceCloneBot

## 项目目标

做一个**本地运行、免费、低资源、支持长文本**的 Telegram 语音回复 bot：

- 朋友新开与 bot 的对话即可使用
- bot 代替 Conan 的风格进行文字/语音回复
- 支持把文本合成为语音消息
- 未来支持语音输入 → 本地识别 → 再用 Conan 声音回复
- 重点是**自然度、稳定性、低延迟**，而不是花哨功能

## 用户诉求

Conan 的核心要求：

1. **本地实现**，不依赖远程大模型 API
2. **免费**，尽量不要引入按量计费服务
3. **时长可做到 2 分钟以上**
4. **资源占用尽量少**
5. **生成速度快**
6. 声音要尽量接近本人，能用于 Telegram 对话场景

## 结论先行

当前优先级建议如下：

1. **Qwen3-TTS**：主线方案，综合最均衡
2. **MOSS-TTS Local / Realtime**：长对话和稳定性备选
3. **OpenVoice V2**：低资源保底方案
4. **F5-TTS**：可用备选，但模型许可不如前面几项理想
5. **Fish Audio S2**：自然度强，但资源与许可不适合作为首选

## 方案评估

### 1. Qwen3-TTS

**定位**：主线方案

**优点**
- 支持本地部署
- 开源，Apache 2.0
- 支持 voice clone
- 支持流式生成
- 有 0.6B / 1.7B 两档，可按机器能力选择
- 适合 Telegram 这种实时交互场景

**适合场景**
- 语音消息回复
- 长文本分段合成
- 先文字后语音的 bot 交互

**备注**
- 优先从 0.6B 起步，质量不够再切 1.7B

### 2. MOSS-TTS Family

**定位**：长文本稳定性与多轮对话备选

**优点**
- 开源，Apache 2.0
- 强调长文本稳定输出
- 有 MOSS-TTS Local / Realtime
- 公开资料里支持 torch-free / llama.cpp / ONNX 路线
- 更偏生产可落地

**适合场景**
- 2 分钟以上语音合成
- 连续多轮对话回复
- 需要更稳定的长句衔接

### 3. OpenVoice V2

**定位**：低资源保底方案

**优点**
- MIT 许可，最省心
- 本地可跑
- 实现门槛较低
- 适合先把端到端链路跑通

**缺点**
- 自然度不如最新一代模型
- 更适合原型或 fallback

### 4. F5-TTS

**定位**：实用备选

**优点**
- 社区成熟
- 支持多风格、多说话人
- 支持 chunk inference

**缺点**
- 模型许可不如 Apache/MIT 路线理想
- 不作为第一选择

### 5. Fish Audio S2

**定位**：自然度强，但不作为首选

**优点**
- 自然度和表现力很强
- 对情绪、风格控制较丰富

**缺点**
- 资源占用更高
- 许可和落地成本不符合“免费、本地、轻量”的目标

## 推荐架构

### 总体链路

Telegram 用户消息
→ 本地接收 bot
→ 判断文本/语音
→ 本地 ASR（如果是语音）
→ 本地 LLM 生成回复文本
→ 本地 TTS 合成 Conan 声音
→ Telegram 语音消息/音频消息回传

### 推荐组件

- **Telegram bot**：`aiogram` 或 `python-telegram-bot`
- **ASR**：`faster-whisper`
- **LLM**：本地量化模型（优先 Qwen 系 3B/7B 级别）
- **TTS**：Qwen3-TTS 主线，MOSS-TTS 备选
- **音频处理**：`ffmpeg`
- **存储**：`SQLite`
- **服务层**：`FastAPI` + worker

## 长文本 / 2 分钟以上的合成策略

不要整段一次性硬生成，建议：

1. 先把回复文本切成句子或短段
2. 每段单独 TTS
3. 保持同一参考音频/同一 voice prompt
4. 段与段之间插入自然停顿
5. 最后拼接成完整音频

这样更稳，也更像真人说话。

## 声音素材准备

### 最少可用
- 3–5 分钟清晰语音

### 更推荐
- 10–20 分钟高质量语音

### 录音要求
- 安静环境
- 同一麦克风
- 统一音量
- 少回声、少背景噪音
- 覆盖多种语气和句式

### 建议准备的参考片段
- 平静
- 轻松
- 正式
- 快语速
- 慢语速

## Telegram 输出形态

### 默认优先
- **语音消息**（OGG/Opus）

### 备选
- **音频文件**（MP3/WAV）

建议默认发语音消息，体验更像真人。

## 分阶段实施

### Phase 1：POC
目标：先跑通最小链路

功能：
- Telegram bot 收文本
- 本地 LLM 回文本
- TTS 合成
- 发回语音消息

验收：
- 能稳定回复
- 声线基本像本人
- 单条消息可正常输出

### Phase 2：语音输入
功能：
- 下载 Telegram 语音
- 本地 ASR 转文字
- 再走同样回复链路

验收：
- 朋友发语音也能聊

### Phase 3：长回复稳定化
功能：
- 分句合成
- 缓存高频回复
- 自动插入停顿
- 控制总时长

验收：
- 2 分钟以上回复不明显崩坏

### Phase 4：风格增强
功能：
- 更贴近 Conan 的措辞
- 按场景选择简短/正式/轻松语气
- 对不同朋友使用不同回复风格

## 风险点

1. **像但不够像**
   - 需要更干净的参考音频
   - 多参考片段
   - 调整分段策略

2. **长文本越说越怪**
   - 必须分句
   - 不能一把梭

3. **延迟偏高**
   - 模型降级到更小版本
   - 缓存常用短句
   - 优先文字再补语音

4. **朋友误解身份**
   - bot 名称和简介要明确
   - 不建议伪装成真人账号

## 推荐落地顺序

1. **先做 Qwen3-TTS 主线 POC**
2. 如果长文本稳定性不够，再接 **MOSS-TTS**
3. 如果机器资源吃紧，再准备 **OpenVoice V2** 作为保底

## 当前状态

- 已完成技术路线调研
- 已确定优先级：Qwen3-TTS > MOSS-TTS > OpenVoice
- 下一步：拆成具体实现任务，开始做 POC

## 备注

这个项目的核心不是“生成一段好听的声音”，而是：

- **像 Conan**
- **说得稳**
- **能长说**
- **本地跑得动**
- **Telegram 里用得顺**
